# VintageWisdom UI 设计文档

## 1. 文档目标

本文档用于统一 VintageWisdom 前端界面的视觉方向、组件语言、页面布局和交互约束，供以下场景共同使用：

- 指导 `frontend/` 中 Next.js 界面的持续迭代
- 作为设计与开发协作时的统一基线
- 作为后续 AI 生成页面、组件和原型时的高质量提示词来源

当前文档参考了 `awesome-design-md-main` 中常见的设计文档结构，吸收了 Linear、Notion、Ollama、Airtable 等案例在“设计系统文档化”上的写法，但视觉上优先服从 VintageWisdom 现有产品定位和已落地界面。

## 2. 产品定位与界面气质

VintageWisdom 不是普通的信息展示站点，而是一个面向“历史案例复用与个人决策支持”的工作台。界面应传达以下感受：

- 深色、冷静、可信，像一个持续在线的分析控制台
- 信息密度较高，但必须有明确层级，不能做成拥挤的数据墙
- 重点不是炫技动画，而是帮助用户快速进入“查看案例 -> 发起查询 -> 理解图谱 -> 调整 AI 配置”的工作流
- 视觉语言要兼顾“知识工具”与“推理系统”气质，避免营销站风格

关键词：

- `Decision cockpit`
- `Historical intelligence`
- `Dense but readable`
- `Dark workspace`
- `Local-first AI`

## 3. 当前设计方向总览

当前前端已经形成一套可延续的深色控制台风格，核心特征如下：

- 整体采用深海军蓝到近黑的渐变背景，而不是纯黑
- 使用紫蓝色作为主要强调色，承担 CTA、焦点和关键状态提示
- 面板采用半透明磨砂质感，配合柔和阴影形成层次
- 卡片、侧边栏、顶部标题区都偏大圆角，形成“高级工作台”视觉
- 使用细网格背景增强“分析系统”氛围，但透明度必须非常低
- 通过字重、字号、留白、微弱边框来管理层级，而不是依赖大量颜色

这意味着文档后续不再沿用最初那版“极简浅色 dashboard prompt”，而是正式将 VintageWisdom 定义为：

**深色、高密度、知识分析型决策工作台。**

## 4. 视觉主题与氛围

### 4.1 整体基调

- 主背景使用冷色深底，避免纯黑造成压迫感
- 页面不追求强对比霓虹感，而是偏“克制发光”
- 主要视觉重心集中在面板、图谱画布、查询区、案例卡片
- 背景允许有少量径向光晕和网格纹理，但只能作为氛围，不可抢内容

### 4.2 风格参考落点

从参考库中，VintageWisdom 更接近以下混合方向：

- 借鉴 `Linear` 的深色控制台层级与高密度信息组织
- 借鉴 `Notion` 的文档式清晰结构与可读性优先原则
- 借鉴 `Ollama` 的克制交互与简化状态表达
- 借鉴 `Airtable` 的工作台组件分区思路

但禁止直接复制任何单一品牌风格。VintageWisdom 必须保持自己的“历史推理系统”辨识度。

## 5. 色彩系统

以下颜色已与当前前端实现保持一致，可直接作为设计 token 使用。

### 5.1 背景与表面

- `--bg-primary: #0f172a`
- `--bg-secondary: #162033`
- `--bg-tertiary: #1f2a3d`
- `--bg-elevated: #334155`
- `--panel-strong: rgba(15, 23, 42, 0.9)`
- `--panel-soft: rgba(15, 23, 42, 0.72)`
- `--panel-muted: rgba(30, 41, 59, 0.74)`

### 5.2 文本

- `--text-primary: #f8fafc`
- `--text-secondary: #cbd5e1`
- `--text-muted: #94a3b8`
- `--text-disabled: #64748b`

### 5.3 强调色

- `--accent-primary: #6366f1`
- `--accent-secondary: #818cf8`
- `--accent-hover: #4f46e5`
- `--accent-glow: rgba(99, 102, 241, 0.34)`

### 5.4 语义状态

- `--success: #22c55e`
- `--warning: #f59e0b`
- `--error: #ef4444`
- `--info: #3b82f6`

### 5.5 边框与阴影

- `--border-subtle: rgba(148, 163, 184, 0.14)`
- `--border-strong: rgba(148, 163, 184, 0.28)`
- `--shadow-soft: 0 20px 55px rgba(2, 6, 23, 0.22)`
- `--shadow-panel: 0 16px 48px rgba(2, 6, 23, 0.28)`

### 5.6 用色原则

- 紫蓝强调色只用于主按钮、焦点、激活导航、关键数据高亮
- 成功/警告/错误只用于状态反馈，不可泛化为装饰色
- 文本层级优先依赖亮度差，而不是增加颜色数量
- 背景层级应通过透明度、模糊、阴影区分，而不是使用大量不同底色

## 6. 字体系统

当前代码中通过 `--font-body`、`--font-heading`、`--font-mono` 管理字体角色，设计上建议固定以下职责：

### 6.1 字体角色

- 正文：现代无衬线，适合长时间阅读
- 标题：比正文更有辨识度，但不能过度装饰
- 等宽：用于 case id、API 地址、快捷键信息、技术元数据

### 6.2 字体层级建议

| 角色 | 建议字号 | 字重 | 行高 | 用途 |
|------|----------|------|------|------|
| Hero Title | 38px-42px | 600 | 1.1 | 首页/核心页主标题 |
| Section Title | 24px-30px | 600 | 1.2 | 页面一级区块标题 |
| Card Title | 18px-20px | 600 | 1.3 | 卡片标题 |
| Body | 14px-16px | 400 | 1.6-1.8 | 常规正文 |
| Meta | 12px-13px | 400-500 | 1.5 | 次级说明、标签、注释 |
| Eyebrow | 11px | 500 | 1.4 | 分区标签、系统标识 |
| Mono Meta | 11px-12px | 400 | 1.4 | id、路径、终端信息 |

### 6.3 字体使用原则

- 标题统一略微负字距，增强凝聚感
- 大段说明文字优先使用 `text-muted`，避免主文本过亮
- `Eyebrow` 一律使用大写或接近全大写风格，并增加字距
- 等宽字体只承担“技术标签”角色，不参与主阅读流

## 7. 间距、圆角与布局系统

### 7.1 间距基线

采用 8px 基础网格，并允许 4px 微调。

推荐节奏：

- 4px：微调、图标与文字间距
- 8px：最小内容间距
- 12px：紧凑组件内边距
- 16px：标准间距
- 20px：舒适卡片内间距
- 24px：模块分组间距
- 28px：大卡片圆角/主要容器留白
- 32px+：页面级留白

### 7.2 圆角体系

- `16px`：主按钮、输入框、辅助按钮
- `20px-24px`：普通卡片、内容容器
- `28px`：页面级大面板、主工作区
- `999px`：徽标、状态 badge、移动端胶囊导航

### 7.3 页面骨架

桌面端推荐采用“三层工作台”结构：

1. 左侧主导航
2. 顶部标题与快捷操作区
3. 主内容区

其中：

- 侧边栏负责模块切换和工作区上下文
- 顶部区域负责页面语义、主 CTA、系统状态摘要
- 主内容区承载卡片栅格、表单、图谱或结果面板

## 8. 组件设计规范

### 8.1 App Shell

对应当前 `AppShell.tsx` 的整体框架。

要求：

- 左侧导航在桌面端固定存在，移动端折叠为顶部胶囊导航
- Header 需要包含 `section / title / subtitle / CTA / quick badges`
- 主内容区使用大圆角包裹，形成“工作台内部画布”
- 页面间统一标题风格，保证切换时的系统连续性

### 8.2 导航

#### 侧边导航

- 默认状态透明或弱底色
- 激活状态使用紫蓝渐变或轻微发光描边
- 每一项应支持“标签 + 序号/缩写”
- hover 只允许轻微提亮和边框显现，不做位移动画

#### 移动导航

- 使用圆角胶囊按钮
- 允许横向滚动
- 激活态要明显，但不能比桌面更重

### 8.3 按钮

#### 主按钮 `vw-btn-primary`

- 使用紫蓝渐变
- 允许轻微上浮和阴影增强
- 用于关键动作，如 `Start Query`、`Save decision log`

#### 次按钮 `vw-btn-secondary`

- 半透明深底 + 细边框
- hover 时边框强调、文字提亮
- 适合次级入口与保守操作

按钮原则：

- 主页面内最多一个最强按钮群
- 相邻按钮权重必须明显分层
- 危险操作不得沿用主按钮样式

### 8.4 卡片

#### 面板卡片 `vw-panel`

- 用于页头区、主容器、重要总结区
- 背景需要带纵向渐变与磨砂感
- 阴影高于普通卡片

#### 普通卡片 `vw-card`

- 用于 KPI、案例列表、说明块、推荐项
- 背景应比 panel 更轻一些
- 与背景之间需要有足够边界，但不应过亮

卡片通用规则：

- 标题区和内容区之间至少有一个明确的垂直节奏
- 卡片内部可再嵌套弱一级容器，但最多嵌套两层
- hover 时以边框提亮为主，避免强位移

### 8.5 输入类组件

包括：

- 查询输入框
- 设置页配置表单
- 文本域
- 下拉选择器

规范：

- 输入类组件统一使用半透明暗底
- focus 时统一出现紫蓝色外圈
- 大文本输入区要足够高，允许用户完整表达情境
- placeholder 只能作为示例提示，不能承担说明文档职责

### 8.6 Badge / 标签

建议延续当前四类：

- `vw-badge-accent`
- `vw-badge-success`
- `vw-badge-warning`
- `vw-badge-error`

用途：

- 领域标签
- 系统状态
- 本地优先 / 键盘就绪等特性标识
- 数量与类型的紧凑表达

禁止：

- 同一区域堆叠过多 badge
- badge 承担长文本说明

### 8.7 数据统计组件

KPI 卡片应遵循：

- 数值最大，说明最简
- 颜色只强调关键指标，不要全卡上色
- 一页内最多 3 到 5 个核心数字
- 同一排卡片的高度和节奏尽量统一

### 8.8 列表与案例卡片

案例列表是核心交互对象之一，必须满足：

- 快速扫描：先看到领域、标题、摘要，再看 id
- 元信息不能喧宾夺主
- 每张卡片可点击面积尽量完整
- hover 用“边框增强 + 轻微上移”即可

建议内容顺序：

1. 领域 badge + case id
2. 标题
3. lesson core 或 description 摘要

### 8.9 查询结果区

查询页建议保持“双栏结构”：

- 左栏：案例召回结果
- 右栏：AI reasoning 与 recommendations

这样符合用户的阅读顺序：

- 先看依据来自哪里
- 再看系统怎么解释
- 最后看该怎么行动

### 8.10 加载与空状态

当前已有的加载页方向是正确的，应继续保持：

- 居中聚焦
- 简短说明系统正在做什么
- 显示当前查询上下文

空状态要求：

- 解释为什么为空
- 给出下一步建议
- 语气应支持性而不是报错式

## 9. 页面级设计要求

### 9.1 Dashboard 首页

目标：

- 让用户在 5 秒内理解当前系统运行状况
- 快速进入查询、案例、导入三个核心操作

首页必须包括：

- 核心价值说明
- 系统脉搏或运行摘要
- KPI 指标
- 最近案例
- 领域分布
- 建议下一步动作

### 9.2 Cases 页面

目标：

- 以高效率浏览历史案例库
- 支持从摘要直接进入详情

需要强调：

- 筛选效率
- 列表扫描效率
- 元数据可见但不拥挤

### 9.3 Query 页面

目标：

- 成为整套系统的主入口之一
- 允许用户用自然语言提出复杂决策问题

页面结构建议固定为三段：

1. 输入引导区
2. 示例 prompts / 输出说明
3. 结果视图

### 9.4 Import 页面

目标：

- 让用户清楚知道支持哪些导入格式
- 理解导入进度、任务状态和异常信息

应包含：

- 文件格式入口
- 导入参数或标签配置
- 任务状态与进度反馈
- 导入结果摘要

### 9.5 Graph 页面

目标：

- 把图谱视为“分析画布”，而不是普通图表

要求：

- 图谱区域优先保证可视性和交互清晰度
- 周边控制面板应退居辅助位置
- 节点、边、标签都要保持高可读性
- 缩放、拖拽、筛选的反馈要即时

### 9.6 Settings 页面

目标：

- 让 AI 运行环境可控、可验证、可切换

必须包括：

- API 状态
- 本地 Ollama / 远程 OpenAI 兼容配置
- 模型名称、API Key、基础提示词等设置
- 配置保存与成功/失败反馈

## 10. 图谱可视化规范

知识图谱是 VintageWisdom 的关键识别点，需要单独规定。

### 10.1 视觉原则

- 背景比普通页面更克制，避免干扰节点识别
- 节点颜色数量控制在少量类别内
- 边应尽量轻，节点才是视觉重点
- 标签只在必要时完全展开，避免满屏文字噪音

### 10.2 节点设计

- 基础形态以圆形为主
- 节点大小可表示权重或连接度，但比例不要过度夸张
- 激活节点可使用发光边缘或更亮描边

### 10.3 边与关系

- 默认细线
- hover 或选中时可增强透明度与亮度
- 若存在聚类，可用很浅的包围色或筛选高亮表达

### 10.4 交互要求

- 鼠标滚轮缩放
- 拖拽节点
- 拖动画布
- 节点点击显示详情
- 支持筛选或聚焦某一类关系

## 11. 动效原则

VintageWisdom 不需要花哨动效，只需要“帮助理解状态变化”。

允许：

- hover 渐变
- 面板轻微浮起
- modal / drawer 滑入
- 查询加载中的旋转反馈
- 结果区渐进显现

禁止：

- 连续抖动
- 大范围元素飞入
- 高频闪烁
- 与任务无关的装饰动画

动效时长建议：

- 150ms：颜色、边框、hover
- 200ms：阴影、轻微位移
- 250ms-300ms：抽屉、弹层

同时必须支持 `prefers-reduced-motion`。

## 12. 响应式规范

### 12.1 断点建议

- Mobile: `< 768px`
- Tablet: `768px - 1279px`
- Desktop: `>= 1280px`

### 12.2 响应式策略

桌面端：

- 保持左侧导航 + 顶部头部 + 多栏内容

平板端：

- 可以缩窄侧边栏
- 双栏逐步转为上下堆叠

移动端：

- 左侧导航隐藏为顶部横向导航
- 所有主要点击区域高度不低于 44px
- 查询输入框、主按钮、结果卡片改为单列流式布局

### 12.3 移动端优先注意事项

- 保证大文本输入的可编辑性
- 图谱页面在移动端应提供简化版交互或辅助说明
- 不要在手机端强行保留桌面大面积对比布局

## 13. 可访问性要求

- 所有交互元素要有清晰 focus 状态
- 文本与背景对比度必须满足基础可读性
- 不能只用颜色表达成功/错误
- 图标按钮必须有文字或 aria 标签
- 加载状态要有可理解的文案提示
- 错误信息不能只写技术报错，必须告诉用户下一步怎么办

## 14. 资源与图标策略

### 14.1 图标

- 优先使用简洁、线性、可读的图标
- 导入、查询、图谱、设置等主功能图标应保持统一风格
- 不要混用多个图标体系

### 14.2 插图与占位资源

如果后续需要 AI 生成原型图、视觉稿或占位资源，应优先要求使用可商用或免版权来源，例如：

- Font Awesome
- Unsplash
- Pexels

但在正式产品界面中，应尽量少用装饰性图片，优先用结构和排版建立气质。

## 15. 前端文件组织建议

结合当前项目结构，推荐继续保持以下职责划分：

```text
frontend/
  src/
    app/              路由页面与全局样式
    components/       可复用界面组件
    lib/              API、图谱与通用逻辑
  public/             静态资源
```

如果后续样式复杂度继续上升，可补充：

```text
frontend/src/
  styles/             主题 token、组件分层样式
  components/ui/      基础 UI 组件
  components/domain/  业务组件
```

## 16. 设计执行清单

后续每次新增页面或大改版时，至少检查以下项目：

1. 是否符合“深色决策工作台”基调
2. 是否复用既有 token，而不是临时新增随机颜色
3. 是否保持标题、卡片、按钮、badge 的统一语言
4. 是否兼顾桌面与移动端
5. 是否提供清晰的加载、空状态、错误状态
6. 是否让用户更快完成案例浏览、查询、导入或配置任务

## 17. 给 AI 的页面生成提示词

以下提示词可用于后续生成新页面、重构视觉稿或让 AI 辅助补全界面：

请遵循以下要求：

1. 整体风格是浅色知识分析控制台，而不是营销官网。
2. 主背景使用冷色深海军蓝到白色渐变，允许非常轻的网格背景和径向光晕。
3. 主视觉 token 参考：
